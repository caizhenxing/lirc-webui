Overview


This page deals with development issues and if you are looking to help develop, maintain or even document XMLProcess then you are my favourite sort of person

Development Areas

There are a number of areas in which development could occur within XMLProcess and it is probably simplest if these are split out as different people have skills in different areas.
Each item is given an ease of implementation and an importance ranking so people can decide to do an easy but low importance item or a high importance item etc.

General

  • Dynamic redeploy using Ant -easy -low
  • Tie into local Search Engine e.g. google desktop. Depends on Dynamic Typing -easy -low
  • Dynamic typing. Check xml against backing files for a match -hard - high
  • Virtual Document. A server could create a Virtual Document served in chunks -hard -low
  • Backing File element handling. How to handle XML element creation such as:
    Choice, order, If element A then no Element B etc. -hard -medium
  • Filter tree. Given a user the ability to filter out only those items they are interested in -medium -medium
  • Suppport IE. Currently only tested to work on Firefox due to JavaScript and CSS differences. Possibly just target IE7 -hard -medium

Java

  • Clean up existing code. Remove dead methods and fields etc -easy -high
  • Optimise code for speed and efficiency. -hard -high
  • JavaDoc. Much of the code needs documenting -easy - high
  • Check with Checkstyle/PMD for correct naming etc. Need to check code with standard code quality tools and then make the changes -easy -high

XML Handling

  • Default namespace handling. Currently the code strips off default namespaces and then re-applies them when the document is saved.
    Could be configurable but should handle default namespaces by creating fake prefixes when required -hard -medium
  • W3C Schema/Relax useage. Currently schemas are not used outside of XForms (for typing) - could be used to build documents -hard -medium

JSF - MyFaces

  • Move to a newer version of MyFaces. The current version is not XML compliant in it's output. For example an input is rendered as
    <input name="jsf_viewid" id="jsf_viewid" value="/XFormMain.jsp" type="hidden"> -medium -high
  • Write a JSF library to make editing XML Elements easier and prettier if a JSFEditor is required. -hard -low
  • Layout - XFormMain. At the moment this is using JSP to do the layout Consider a more flexible route e.g. portlets etc ? -hard -low
  • Use of Facelets. Mostly to trap JSP errors but also possibly for use in layout.

Chiba - XForms

  • Move to Chiba 2.0.
    Requires Java 1.5.
    Means rewriting XFormHTMLProcessor
    Many advantages -easy -high
  • Look at how best to externalise Chiba so that newer versions can be integrated easily. -medium -high
  • If moving to Chiba 2.0 check that the XForms mediatype="text/html" is handled and that FCK is OK -easy -high
  • See if there is a way of detecting a native XForms handler. -hard -low

Eclipse

  • Package up application as a plugin. See here for details. Dependant upon the Dynamic Typing or an eclipse class which can do that -medium -high

FileIO

  • Create new FileIO handlers including the Java Classes and the JSF Pages.  The architecture allows for any file source/sink but only file is implemented at the moment.
    Ideas include:
    • Subversion
    • CVS
    • JMS
    • WebService
    • FTP
    • SQL Database
    • JNDI
